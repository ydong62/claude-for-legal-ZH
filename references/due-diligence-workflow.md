# 尽职调查工作流方法论

> 共享参考文件——corporate-legal 插件的尽调类技能使用本方法论。
> 目标：把尽职调查从一次性问答变成可持续推进的项目档案。

---

## 总目标

- 原始底稿可追溯
- AI 可读版本可复用
- 事实查明可增量更新
- 风险条线可持续积累
- 法律研究可验证
- 输出文件可交付

---

## 六阶段流程

### 阶段 1：项目立项

新项目开始时，先建立或补齐项目元数据：

**项目画像**（至少包含）：
- 项目名称、客户或目标公司
- 事项类型（并购/融资/上市/合规/其他）
- 交易类型、行业、法域
- 业务模式、特殊场景（数据合规/AI 合规/国企合规等）
- 预期交付物、保密要求

**初始文件**：
- 项目画像（project profile）
- 会话日志（session log）
- 材料索引（material index）

### 阶段 2：指引加载

根据项目画像加载尽调清单：

1. **通用尽调清单**：适用于所有项目的基准检查项
2. **行业专项清单**：根据目标公司行业加载（制造/房地产/金融/科技/医疗等）
3. **交易类型清单**：根据交易性质加载（股权收购/资产收购/合并/少数股权投资等）
4. **特殊场景清单**：数据合规、AI 合规、国企合规、环保合规、劳动用工等

加载结果记录到项目风险清单（project risk checklist），作为后续事实查明的导航框架。

### 阶段 3：底稿摄入与转换

用户提供文件、文件夹或链接后：

1. **识别文件性质**：区分合同、证照、审批文件、财务报表、诉讼材料、知识产权证书等
2. **判断事实价值**：决定是否纳入正式底稿范围
3. **分类归档**：按文件类型和涉及主体分类
4. **生成 AI 可读版本**：将扫描件、图片、非标准格式转换为结构化文本
5. **更新索引**：同步更新材料索引和底稿清单

### 阶段 4：事实查明与证据映射

围绕底稿和风险清单，持续维护：

| 文件 | 内容 | 规则 |
|------|------|------|
| 事实登记册 | 跨条线的核心事实，每条含事实编号、表述、支持材料、可信度、影响的风险点 | 新增材料必须校准既有事实，而非仅做摘要 |
| 证据映射 | 每个核查事项对应的事实依据、底稿依据、法规依据 | 无证据链的内容只能进入待核实事项 |
| 矛盾日志 | 材料之间、材料与访谈之间、事实与法律判断之间的冲突 | 冲突不得被自动抹平 |
| 未决事实 | 待核实、待补充材料或待访谈确认的事实 | 持续追踪至关闭 |

**证据链格式**（每个风险/法律结论应能拆成）：
```text
风险编号 → 核查事项 → 初步结论 → 事实依据 → 底稿依据 →
法规依据 → 相反证据 → 证据缺口 → 风险判断 → 整改建议 → 复核状态
```

### 阶段 5：问题条线推进

根据行业、交易类型和风险清单建立若干问题条线。每条线对应一个独立的法律/合规维度（如：股权结构与历史沿革、重大合同、知识产权、劳动用工、税务、环境、数据合规等）。

**条线来源**：
- 通用清单重点风险
- 行业清单重点风险
- 交易类型清单重点风险
- 用户特别关注事项
- 新增底稿暴露出的重大风险

**每个条线建议结构**：
```text
条线文件夹/
  facts.md              ← 已查明事实
  materials.md           ← 相关材料索引
  legal_research.md      ← 法律研究结果
  risk_analysis.md       ← 风险分析
  gap_list.md            ← 证据缺口
  interview_questions.md ← 访谈问题
  supplement_request.md  ← 补充材料要求
  report_section.md      ← 报告章节草稿
```

条线不是越多越好——每个条线应能对应事实查明和报告章节。

### 阶段 6：交付输出

**输出原则上来自问题条线汇总，不重新从零生成。**

默认交付物：
1. **尽职调查报告**：按"调查事实描述 → 法律评价分析 → 提出意见或建议"三段式组织每板块
2. **补充材料清单**：未获取的底稿、待补充的文件
3. **访谈清单**：需向管理层/业务部门核实的问题
4. **整改清单**：已发现的合规缺陷及建议整改方案
5. **底稿目录**：全部已审阅材料的索引
6. **内部未决问题清单**：律师团队内部待讨论事项

## 报告写作方法论

### 体例结构

```
目录 → 释义 → 引言 → 摘要 → 正文 → 附件
```

**引言**至少包含四项：尽调目的、调查工作情况、特别提示（前提假设）、免责声明。

**正文**按尽调内容分为若干板块，每板块三段式：
1. **调查事实描述**：事实性描述，不涉及法律评价
2. **法律评价分析**：对尽调结果进行法律分析并作出评价
3. **提出意见或建议**：给出预防/减轻/规避风险的确定性意见或参考性建议

**即使相关事项不存在法律风险，也应给出明确评价**——避免产生"到底有没有法律风险"的疑问。

### 风险揭示分级

- 对**可解决或降低的风险**：提出具体方案
- 对**无法全部避免但不影响交易目的的风险**：作出提示即可，不用过度强调
- 对**可能影响交易目的实现的重大风险**：充分说明并给出防范建议

### 客户导向写作

以客户为中心：先呈现事实，再列举法律，最后给出评价。

```
小前提（法律事实）→ 大前提（法律规定）→ 结论（法律评价 + 风险提示 + 防范建议）
```

## 质量门禁

### 客户可见 vs 内部版本

- **内部工作底稿**：可包含 AI 转换说明、待核实事项、内部判断过程
- **客户可发送版本**：不得包含提示词、工具日志、内部 AI 过程语言、未经复核的推断

### 输出前检查

- [ ] 每个重大风险是否有底稿依据
- [ ] 每个法律结论是否有法规或权威来源
- [ ] 是否列明证据缺口
- [ ] 是否区分事实、法律、推理和建议
- [ ] 是否处理新增底稿带来的矛盾
- [ ] 是否去除内部过程语言
- [ ] 是否符合用户指定格式
- [ ] 文件名是否清晰、可追溯

### 依据不足时的处理

如果依据不足，应输出：
- 缺失材料
- 待访谈问题
- 暂不能形成结论的原因
- 建议补充路径

**不得用流畅文字掩盖证据不足。**
