# 庭审准备框架

> 共享参考文件——litigation-legal 插件的庭审准备技能使用本框架。
> 适用于诉讼/仲裁庭审准备，兼顾原告/被告/仲裁员视角。

---

## 案件目录结构

每个案件独立文件夹，全流程隔离：

```
{案件简称}/
├── input/          ← 原始材料（起诉状/申请书、答辩状、证据目录等）
├── scratch/        ← 中间产物（分析笔记、法律检索记录、草稿）
└── output/         ← 最终交付物（庭审提纲、争议焦点归纳等）
```

**案件简称命名**（优先级从高到低）：
1. 用户提供案件名称
2. 案号
3. 当事人 + 案由（如"A 公司诉 B 公司买卖合同纠纷"）

**input/ 只读纪律**：材料存入后不在 input/ 内修改。

---

## 案件处理流程

### Step 1：材料收集

- 用户通过对话描述案情 → 整理为结构化摘要
- 用户提供文件路径或直接发送文件 → 存入 `input/`
- 不完整材料标注缺口，不猜测补充

### Step 2：案件分析

1. **梳理基础信息**：当事人、诉讼/仲裁请求、案由、管辖依据、程序阶段
2. **时间线整理**：按时间线组织事实
3. **法律问题识别**：归纳本案涉及的法律问题和争议焦点
4. **法律检索**：按知识库路由协议（`references/knowledge-base-crossref.md`）检索相关法律依据、类案和实务观点
5. **补充检索**：知识库不足时通过已配置的 MCP 连接器补充检索
6. **分析记录**存入 `scratch/`

### Step 3：输出庭审提纲

---

## 庭审提纲默认内容

无特别要求时，庭审提纲默认包含以下五个模块：

### 1. 案件概览

- 当事人信息
- 诉讼/仲裁请求摘要
- 程序信息（受理机构、案号、受理日期、程序阶段）

### 2. 争议焦点归纳

- 逐项列出核心争议点
- 每个争议点附：各方主张、关键证据、可能的法律适用

### 3. 事实查明提纲

- 需要查明的事实清单
- 对应证据编号和待确认事项
- 标注已确认事实 / 争议事实 / 待查事实

### 4. 法律适用提纲

- 各争议焦点涉及的法律依据（附具体条文编号）
- 关键法条摘录
- 类案裁判倾向（如有）
- 举证责任分配分析

### 5. 庭审发问提纲

- 按争议焦点分组
- 针对各方当事人的发问清单
- 发问目的说明（证明什么事实）

## 证据状况标注

审查证据时使用统一标注：

| 标记 | 含义 |
|------|------|
| ✅ | 事实可证明，证据充分 |
| ⚠️ | 事实有证据但证明力待评估 |
| ❌ | 事实主张但缺乏证据 |
| 🔍 | 需通过申请调取、鉴定、证人等方式补强 |

---

## 庭前准备清单

```
□ 事实时间线整理完毕
□ 证据清单及举证顺序确定
□ 对方可能抗辩及应对预案
□ 关键法律依据标注（含具体条文）
□ 需要庭审发问的问题清单
□ 鉴定/调查取证/追加当事人等申请是否已提交
□ 模拟庭审要点记录（如有条件）
```

## 角色适配

本框架适用于以下角色视角，各插件技能根据实务画像自动适配：

| 角色 | 侧重点 |
|------|--------|
| **原告/申请人方** | 证明请求权构成要件、组织进攻性证据、准备对方抗辩应对 |
| **被告/被申请人方** | 否定构成要件、组织防守性证据、寻找程序抗辩（管辖/时效/主体） |
| **仲裁员/法官** | 兼顾双方观点、关注证据采信和法律适用、服务庭审主持 |
| **代理律师** | 最强法律主张 + 最稳妥实务路径双轨分析 |

## 案件隔离纪律

1. **严格隔离**：不同案件的材料、分析、输出互不混入
2. **新案确认**：用户提到新案件时，先确认是续讨论旧案还是开新案
3. **路径定位**：开始工作时先确认当前案件文件夹路径，后续操作均基于该路径
4. **跨案引用**：不主动引用其他案件的分析结论，除非用户明确要求
